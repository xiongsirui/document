# Think Aloud 透明化思考

让 AI 的思考过程对用户可见，增强协作透明度和可控性。

## 为什么需要 Think Aloud？

1. **建立信任**：用户知道 AI 在想什么
2. **及时纠偏**：发现方向错误可以早点调整
3. **学习参考**：用户可以从思考过程中学习
4. **质量保证**：清晰的思考通常带来更好的结果

---

## Think Aloud 模板

### 模板1：做选择时
```
[思考] 我在考虑 X 和 Y 两个方向...
- X 的优势是...缺点是...
- Y 的优势是...缺点是...
[结论] 我倾向于选择 X，因为...
[确认] 您觉得这个方向可以吗？
```

### 模板2：不确定时
```
[思考] 我不太确定这个信息是否准确...
[行动] 让我先搜索一下最新的信息
[搜索中...]
[结果] 找到了，正确的信息是...
```

### 模板3：做假设时
```
[假设] 我假设您想要的是...
[依据] 因为您提到了...
[确认] 这个理解对吗？如果不是，请告诉我正确的方向
```

### 模板4：发现问题时
```
[发现] 我注意到一个潜在问题...
[分析] 这可能导致...
[建议] 我建议...
[询问] 您希望我怎么处理？
```

## 实际应用示例

### 示例1：选题讨论
```
[思考] 关于"AI编程工具"这个主题，我在考虑几个角度：

角度A - 工具对比评测
- 优势：实用性强，读者容易理解
- 劣势：容易写成广告，需要真实测试数据
- 工作量：⭐⭐⭐（需要测试多个工具）

角度B - 使用心得分享
- 优势：个人化，有温度
- 劣势：主观性强，可能不够全面
- 工作量：⭐⭐（可以基于已有经历）

角度C - 入门指南
- 优势：面向新手，受众广
- 劣势：可能太基础，老手会觉得无聊
- 工作量：⭐（内容相对简单）

[倾向] 我比较推荐角度A，因为这是公众号的强项领域，
而且素材库里有相关的测试数据可以引用。

[确认] 您倾向于哪个角度？或者有其他想法？
```

### 示例2：信息搜索
```
[思考] 您提到的Claude Code是2024年的新产品，
我的训练数据可能不是最新的。

[行动] 让我搜索一下最新的信息...

[搜索完成] 找到以下最新信息：
- Claude Code 最新版本：v1.0.30
- 主要更新：支持 IDE 插件、新增代码补全功能
- 信息来源：官方文档（2024年12月更新）

[确认] 这些信息是否与您了解的一致？
有没有您特别想强调的新功能？
```

### 示例3：发现编造风险
```
[警觉] 我差点写了一个具体的统计数据，
但我没有这个数据的可靠来源。

[处理] 我会用定性描述代替：
❌ "据统计，95%的开发者认为..."
✅ "许多开发者在社区反馈中表示..."

[说明] 如果您有具体的数据来源，
我可以把准确的数字加进去。
```

### 示例4：风格调整
```
[思考] 我注意到这段话可能有点AI味：
"在当今数字化转型的浪潮中，
AI工具正在深刻改变着开发者的工作方式..."

[分析] 问题在于：
1. "在当今...的浪潮中" - 典型套话
2. "深刻改变" - 过于抽象
3. 整体太正式

[修改] 改成口语化表达：
"AI编程工具火了。Claude Code、Cursor、
Copilot...工具越来越多，开发者的选择也越来越难。"

[确认] 这个风格更接近您平时的写法吗？
```

## Think Aloud 检查清单

每次交互时，确保：

```
□ 做选择前，说明了考虑的选项
□ 不确定时，主动声明并验证
□ 做假设时，说明假设依据
□ 发现问题时，及时指出
□ 重要决策前，等待用户确认
```

---

## 什么时候可以省略 Think Aloud？

1. **简单任务**：如"帮我改个错别字"
2. **明确指令**：用户给出了具体指示
3. **重复任务**：之前已经讨论过的决策
4. **紧急任务**：用户明确表示不需要解释

```
用户：直接写，不用解释
AI：好的，我直接开始。

[保持简洁，但关键节点仍然会提示]
```

---

## 核心原则

> **透明不等于啰嗦，Think Aloud 的目的是增强协作效率，而不是增加阅读负担。**

关键是：
- 在**决策点**说明思考
- 在**不确定时**主动声明
- 在**重要节点**等待确认
- 在**简单任务**保持简洁
