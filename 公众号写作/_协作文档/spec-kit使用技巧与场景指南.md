# Spec-Kit：跟AI编程，你得先学会"说人话"

## 核心价值

这篇文章解决一个痛点：**你用Claude Code、Cursor、Copilot写代码时，是不是经常发现AI理解偏了你的意图？**

Spec-Kit是GitHub开源的一套工具链，它把"随口一说"的需求变成结构化的规格文档，让AI编程更靠谱。

**适合谁看**：正在用AI编程工具（Claude Code、Cursor、GitHub Copilot）的开发者

---

## 先说结论

我用了一段时间Spec-Kit，最大的感受是：**它不是让AI更聪明，而是让你学会"说人话"**。

很多人抱怨AI编程不好用，问题往往不在AI，在需求表达太模糊。"帮我写个登录功能"这种话，AI能理解才怪。Spec-Kit的核心价值是逼你把需求说清楚——写成规格文档，然后AI照着规格干活，出错的概率就小多了。

---

## Spec-Kit是什么？

GitHub在2025年9月开源的工具，全称是"spec-driven development toolkit"。

**先写规格，再写代码。** 核心理念就这么简单。

四个阶段，每个阶段有明确的检查点：
- **Specify** - 定义你要做什么（关注用户旅程和成功标准）
- **Plan** - 规划怎么做（技术栈、架构、约束条件）
- **Tasks** - 拆分成可测试的小任务
- **Implement** - 执行实现

这个流程不新鲜，但Spec-Kit把它和AI编程工具深度整合了——支持GitHub Copilot、Claude Code和Gemini CLI。

## 实际使用场景

### 场景一：新项目启动

以前开新项目，我习惯直接跟AI说"帮我搭个React项目"。结果？AI给我生成一堆我用不上的配置，目录结构乱七八糟。

用Spec-Kit的方式，先跑个命令：

```bash
uvx --from git+https://github.com/github/spec-kit.git specify init my-project
```

然后它会引导你填写：
- 项目要解决什么问题
- 技术栈选型
- 核心功能模块
- 预期的项目结构

这些信息会生成一个`spec.md`文件。接下来不管你用哪个AI工具，都指向这个文件——AI就知道你要什么了。

### 场景二：复杂功能开发

举个例子，要实现一个"用户权限系统"。

以前的做法：直接跟AI说"帮我写个权限系统"，然后AI给你生成一堆代码，跑起来发现根本不符合业务需求，改来改去，最后四不像。

用Spec-Kit，我会先定义功能规格：
1. 用`specify feature`创建一个权限功能的规格文件
2. 填写权限模型（RBAC还是ABAC？）
3. 列出需要控制的资源和操作
4. 说明边界条件（游客能看什么？管理员能改什么？）

这个规格文档写完，问题其实已经想透了。再让AI写代码，基本一次就能用。

### 场景三：多人协作

我们团队里有人用Claude Code，有人用Cursor，还有人坚持用Copilot。

问题很明显：每个人跟AI说话的方式不一样，写出来的代码风格也乱七八糟，每个人维护自己那块时都头疼。

Spec-Kit的思路很简单：**规格文档是唯一真实来源**。

不管谁写代码，都先看`spec.md`。AI工具也读同一个文件，这样大家产出的代码至少在架构层面是一致的。

### 场景四：重构老项目

接手别人的老项目，代码乱得没法看。

Spec-Kit有个`specify analyze`命令挺好用。它能：
1. 分析现有代码结构
2. 生成规格文档
3. 识别可以重构的模块

有了规格文档，再让AI帮忙重构，至少不会改得面目全非。

## 几个实用技巧

### 技巧一：规格文档别写太细

很多人把规格文档写成伪代码，没必要。

我的经验：**说清楚"要什么"，别管"怎么做"**。

举个例子：
- 写：用户登录成功后跳转到首页
- 别写：调用API后，判断status===200，然后router.push('/home')

后者是AI的事，你写太细反而限制了AI发挥。

### 技巧二：用AI帮你写规格

这听起来有点绕——用AI帮你写给AI看的规格？

但真的很好用。你先跟AI聊聊你的需求，让它帮你整理成结构化的文档。这个聊天的过程本身就是一次需求澄清。

### 技巧三：规格文档是活的

别指望一次写完美。我的习惯是：
1. 先写个粗糙版本
2. 让AI生成代码
3. 发现问题，再更新规格
4. 继续迭代

规格文档不是签完就扔的合同，而是会随着项目演进的活文档。

### 技巧四：配合CLAUDE.md

如果你用Claude Code，Spec-Kit的规格文档和项目的CLAUDE.md是绝配：
- `spec.md` - 定义"做什么"
- `CLAUDE.md` - 定义"怎么做"（代码风格、技术偏好）

两个文件配合，AI写出来的代码更像你自己写的一样。

## 什么时候不需要？

老实说，简单脚本、快速原型、改几行代码，直接跟AI说就行。

**Spec-Kit最适合的三个场景**（来自GitHub官方）：

1. **从零开始的新项目（Greenfield）**
   - 写个规格、定个计划，让AI别给你生成一堆通用模板

2. **给现有系统加功能**
   - 这是最适合的场景。先写清楚新功能怎么和现有系统交互，计划里把架构约束写进去，新代码就不会像个"补丁"

3. **老系统重构**
   - 把核心业务逻辑抓出来写成规格，设计新架构，让AI从零重写，不带历史包袱

## 安装和快速上手

```bash
# 安装并初始化（需要先装uv）
uvx --from git+https://github.com/github/spec-kit.git specify init <PROJECT_NAME>
```

然后会生成一个项目结构：
```
my-project/
├── spec.md          # 主规格文档
├── specs/           # 功能规格目录
│   └── features/
├── plans/           # 实现计划
└── tasks/           # 任务列表
```

接下来就是填空题了。

## 写在最后

用了Spec-Kit一段时间后，我有个体会：最大的收获不是AI写的代码更好了，而是**我自己想问题更清楚了**。

写规格文档的过程，本质上是强迫你把模糊的想法变成清晰的描述。这个步骤省不掉，省掉了后面就会翻车。

AI只是执行者，你才是架构师。Spec-Kit让你把架构师这个角色扮演好。

## 总结

Spec-Kit的核心价值：
1. **结构化需求** - 把"随口一说"变成文档
2. **跨工具兼容** - Claude Code、Cursor、Copilot都能用
3. **团队协作** - 统一的规格，一致的产出
4. **需求澄清** - 写规格的过程就是想清楚的过程

如果你经常被AI生成的代码气到，试试Spec-Kit。说不定你会发现，问题不在AI，是你说的不够清楚。
