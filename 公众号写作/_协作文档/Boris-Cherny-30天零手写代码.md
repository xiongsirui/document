# Claude Code 之父实测：30 天零手写代码的 AI 工作流

---

## 什么概念？

1 月 3 日，Boris Cherny 在 X 上发了个帖子，瞬间引爆了整个开发者社区。

Boris 是谁？Claude Code 的创造者，Anthropic 的工程师。

他在帖子里说了一件事：**过去 30 天，他 100% 的代码都是 AI 写的。**

不是"大部分"，不是"辅助"，是**每一行代码**。

更夸张的是，这 30 天他合并了 **259 个 Pull Request**。

换算一下：平均每天 8-9 个 PR。一个普通工程师一周也就出这么几个。

开发者社区炸了。有人说这感觉像在玩《星际争霸》，不再是敲代码，是指挥军队。

---

## 他的工作流长什么样？

### 一、5 个并行终端

Boris 不像我们这样，开一个 Claude Code 窗口，写完一个功能再写下一个。

他同时开 **5 个 Claude Code 终端会话**，编号 1-5。

- 第 1 个在跑测试套件
- 第 2 个在重构旧模块
- 第 3 个在写文档
- 第 4 个在修 Bug
- 第 5 个在优化性能

他怎么知道哪个需要他输入？系统通知。

当一个 Claude 完成任务或遇到问题时，会弹通知。他切换过去，看一下，给个指令，又切到下一个。

除此之外，他还在浏览器里开 **5-10 个 claude.ai/code 会话**，用 `--teleport` 命令在本地和网页之间来回切换。

早上从手机上启动几个会话，到公司后接手继续干。

这不是在写代码，这是在**指挥一支 AI 工程师团队**。

### 二、全用 Opus 4.5

这点挺反直觉的。

现在大家都追求快，用小模型、便宜模型。

Boris 倒好，**全用 Opus 4.5 + thinking 模式**。

这是 Anthropic 最重、最慢、最贵的模型。

他的理由是：虽然它慢，但你不用怎么管它。它工具用得好，几乎不出错，最后反而更快。

这让我想起一句话：**便宜的才是最贵的**。

用小模型省了几秒钟，后面花半小时改 bug。用大模型多花了几秒钟，后面不用改。

这就是"计算税"和"修正税"的权衡。

### 三、CLAUDE.md 作为团队基础设施

Boris 的团队维护一个文件：`CLAUDE.md`，放在 git 仓库里。

每次 Claude 做错什么，他们就往里面加一条规则。

"别再用 X 模式了，我们用 Y 模式。"

"测试文件要放在 `__tests__` 文件夹下。"

"API 响应必须包含 `timestamp` 字段。"

久而久之，Claude 越用越聪明，因为它"记住"了团队的所有约定。

代码审查时，如果发现 Claude 犯了错，他们不只是修代码，还让 Claude 自己更新 CLAUDE.md。

**每个错误都变成一条永久规则。**

这就是"复利工程"：一次投入，永久收益。

### 四、Plan Mode 是标配

Boris 所有非平凡的任务，都从 Plan Mode 开始（Shift+Tab 两下）。

他会和 Claude 来回聊，直到计划满意了，才切换到自动执行模式。

然后 Claude 通常一次性就把活干完了。

很多人跳过计划这步，想"省时间"，结果后面花更多时间修 bug。

**Plan Mode 不是辅助轮，是量两次再切。**

### 五、Slash Commands 和 Subagents

Boris 每天用几十次的命令：`/commit-push-pr`。

写代码、测试、提交、推送、开 PR，一条命令搞定。

这些命令都放在 `.claude/commands/` 文件夹下，用 bash 脚本预计算上下文（git status 等），避免来回和模型对话。

他还用几个 Subagents：

- `__code-simplifier__`：简化代码
- `__verify-app__`：端到端测试
- `__build-validator__`：构建验证
- `__code-architect__`：代码架构
- `__oncall-guide__`：运维指南

模式很简单：**识别重复工作流，把它代码化。**

### 六、给 Claude 验证自己的能力

这是 Boris 说的**最重要的一点**。

> 给 Claude 一个验证自己工作的方法，最终结果质量会提升 2-3 倍。

Claude 每次改动 claude.ai/code，都会用 Chrome 扩展打开浏览器，测试 UI，反复迭代，直到代码能用、体验好。

验证的方式因任务而异：

- **简单**：跑个 bash 命令
- **中等**：跑测试套件
- **复杂**：在浏览器或手机模拟器里测试

**Claude 不只是写代码，它证明代码能用。**

## 这套工作流告诉我们什么？

### 一、并行化胜过优化

Boris 的设置其实一点都不复杂。

没有花里胡哨的配置，没有 clever hacks。

就是**多个简单会话，打败一个复杂会话**。

5 个并行会话，每个做一件事，比 1 个会话做 5 件事，效率高太多。

### 二、工具已经准备好了

很多人还在等"更好的 AI 工具"。

Boris 证明：**工具已经足够好了，问题是你怎么用。**

Claude Code 就在那里，免费试用，开箱即用。

Boris 本人都不怎么自定义配置，"vanilla setup"是他自己的说法。

### 三、思维的转变才是关键

从"AI 是助手"到"AI 是劳动力"。

这不是语义游戏，这是根本性的思维转变。

助手：我让你帮我干活。

劳动力：我管理你来干活。

前者还在敲代码，只是敲得快了点。

后者已经不敲代码了，他在指挥一支 AI 工程师团队。

---

## 我们能学到什么？

不一定每个人都要像 Boris 那样开 10 个并行会话。

但这几点值得思考：

**1. 试试并行会话**

不用 5 个，先从 2 个开始。

一个写功能，一个写测试。或者一个修 bug，一个写文档。

你会发现效率完全不一样。

**2. 写个 CLAUDE.md**

把你项目的约定、规范、偏好，写到一个文件里。

每次 Claude 犯错，就加一条。

一个月后，你会发现 Claude 变聪明了。

**3. 用 Plan Mode**

别跳过计划这步。

花 5 分钟聊清楚要做什么，比花 50 分钟改 bug 强。

**4. 给 Claude 验证的能力**

无论你是跑测试、打开浏览器、还是手动验证。

让 Claude 自己验证自己的工作，质量会提升 2-3 倍。

**5. 别用小模型省那几秒**

用大模型，让它自己干，你不用管。

最后反而更快。

## 最后

Boris 的这套工作流，让我想起一句话：

> "The best tool is the one you don't have to think about."

Claude Code 对 Boris 来说，就是这样。

它不是一个"产品"，它是一个"Unix 工具"。

就像 ls、grep、awk 一样，自然、流畅、无处不在。

30 天零手写代码，听起来很夸张。

但对 Boris 来说，这可能只是新常态。
